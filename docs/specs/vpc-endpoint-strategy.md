# VPC Endpoint Strategy for `awsx.ec2.Vpc`

Status: Draft

## Summary

Add `vpcEndpointStrategy` to `awsx.ec2.Vpc` and keep `vpcEndpointSpecs` as the
single endpoint configuration surface.

`vpcEndpointStrategy` controls whether endpoint specs are treated as raw legacy
pass-through inputs or whether AWSX fills omitted values from resources created
by the same VPC component.

```ts
new awsx.ec2.Vpc("app", {
  vpcEndpointStrategy: "Auto",
  vpcEndpointSpecs: [
    {
      serviceName: `com.amazonaws.${region}.s3`,
      // vpcEndpointType omitted: provider default is Gateway.
      // routeTableIds omitted: AWSX selects generated route tables.
    },
    {
      serviceName: `com.amazonaws.${region}.sqs`,
      vpcEndpointType: "Interface",
      // subnetIds omitted: AWSX selects generated subnets.
      // securityGroupIds omitted: AWSX creates an endpoint SG.
      // privateDnsEnabled omitted: AWSX enables private DNS.
    },
  ],
});
```

The default strategy is `"Legacy"` to preserve current behavior for existing
programs. Existing working `vpcEndpointSpecs` programs do not need to migrate.

## Current Usability

`vpcEndpointSpecs` works today when users already know the IDs needed by the
underlying endpoint resource:

```ts
new awsx.ec2.Vpc("app", {
  vpcEndpointSpecs: [
    {
      serviceName: `com.amazonaws.${region}.s3`,
      vpcEndpointType: "Gateway",
      routeTableIds: knownRouteTableIds,
    },
    {
      serviceName: `com.amazonaws.${region}.sqs`,
      vpcEndpointType: "Interface",
      subnetIds: knownSubnetIds,
      securityGroupIds: [knownSecurityGroupId],
      privateDnsEnabled: true,
    },
  ],
});
```

It is awkward when the natural subnet IDs or route table IDs are generated by
the same `awsx.ec2.Vpc` instance. The caller cannot reference
`vpc.privateSubnetIds`, `vpc.publicSubnetIds`, `vpc.isolatedSubnetIds`, or
`vpc.routeTables` while constructing that same VPC's input object.

Users can still express every endpoint shape in one Pulumi program by creating
lower-level endpoint resources after the VPC:

```ts
const vpc = new awsx.ec2.Vpc("app", {});

const endpointSecurityGroup = new aws.ec2.SecurityGroup("sqs-endpoint", {
  vpcId: vpc.vpcId,
});

new aws.ec2.VpcEndpoint("sqs", {
  vpcId: vpc.vpcId,
  serviceName: `com.amazonaws.${region}.sqs`,
  vpcEndpointType: "Interface",
  subnetIds: vpc.privateSubnetIds,
  privateDnsEnabled: true,
  securityGroupIds: [endpointSecurityGroup.id],
});
```

That workaround remains valid, but the VPC component should be able to fill the
simple generated subnet and route table cases itself.

## Goals

- Preserve current `vpcEndpointSpecs` behavior by default.
- Avoid introducing a second endpoint configuration API.
- Let users opt into AWSX-managed endpoint defaults with one explicit strategy
  switch.
- Fill omitted Interface endpoint subnet IDs from generated VPC subnets.
- Fill omitted Gateway endpoint route table IDs from generated VPC route tables.
- Make common Interface endpoints usable by default when opted into Auto.
- Preserve explicit user-supplied endpoint fields.
- Warn on clearly incomplete Legacy endpoint specs without changing their
  behavior.

## Non-Goals

- Do not remove or deprecate `vpcEndpointSpecs`.
- Do not add a new typed `endpoints` property.
- Do not automatically migrate users to Auto.
- Do not attempt to make Gateway Load Balancer, Resource, or Service Network
  endpoints fully managed in this change.
- Do not add a service catalog or service bundles such as ECR endpoint sets.
- Do not require a separate Pulumi update to discover AWSX-created IDs.

## Public API

Add a new optional VPC input:

```ts
type VpcEndpointStrategy = "Legacy" | "Auto";

interface VpcArgs {
  vpcEndpointStrategy?: VpcEndpointStrategy;
  vpcEndpointSpecs?: VpcEndpointSpecInputs[];
}
```

Behavior:

- Omitted `vpcEndpointStrategy` means `"Legacy"`.
- `"Legacy"` preserves current pass-through behavior.
- `"Auto"` keeps the existing `vpcEndpointSpecs` shape, but fills omitted fields
  where AWSX has a stable component-owned default.

This should be schema-backed and generated for all SDKs. Do not add
TypeScript-only public shapes.

## Strategy Behavior

### Legacy

Legacy is the compatibility mode and the default.

In Legacy mode, AWSX creates one `aws.ec2.VpcEndpoint` for each spec using the
same child logical name and parentage as today. AWSX passes through the spec's
fields and does not infer generated subnet IDs, generated route table IDs,
private DNS, or security groups.

Legacy should emit warnings for specs that are clearly incomplete:

- `vpcEndpointType: "Interface"` with neither `subnetIds` nor
  `subnetConfigurations`.
- `vpcEndpointType: "Gateway"` with no `routeTableIds`.
- Omitted `vpcEndpointType` with no `routeTableIds`, because the provider
  default endpoint type is Gateway.

If `vpcEndpointType` is an unknown `Input<string>`, AWSX should not warn because
the component cannot know which endpoint requirements apply at construction
time.

Warnings should point users to `vpcEndpointStrategy: "Auto"` when they want AWSX
to use generated subnets or route tables. Warnings should not appear for specs
that provide explicit IDs, because those are valid legacy configurations.

### Auto

Auto is opt-in behavior for users who want AWSX to complete endpoint specs from
the VPC resources it creates.

Auto preserves explicit fields. AWSX only fills omitted values.

Auto is type-aware and must know each endpoint type at construction time.
Pulumi child resources cannot be conditionally created inside an `apply`, so
AWSX cannot wait for `vpcEndpointType` to resolve before deciding whether to
infer Interface subnet IDs, create an Interface endpoint security group, or infer
Gateway route table IDs.

In Auto mode:

- Omitted `vpcEndpointType` is treated as Gateway, matching the provider
  default.
- Literal string values such as `"Interface"`, `"Gateway"`,
  `"GatewayLoadBalancer"`, `"Resource"`, and `"ServiceNetwork"` are supported.
- Unknown `Input<string>` values for `vpcEndpointType` are not supported. AWSX
  should fail early with a clear error instead of silently falling back to
  Legacy behavior.

Users with dynamic endpoint types should use `vpcEndpointStrategy: "Legacy"` or
create lower-level `aws.ec2.VpcEndpoint` resources after the VPC.

#### Interface Endpoints

For specs with `vpcEndpointType: "Interface"`:

- If `subnetIds` and `subnetConfigurations` are both omitted, AWSX selects
  generated VPC subnets.
- Subnet selection is one subnet per Availability Zone.
- Selection prefers Private subnets, then Isolated subnets, then Public subnets.
- If `privateDnsEnabled` is omitted, AWSX sets it to `true`.
- If `securityGroupIds` is omitted, AWSX creates one endpoint security group and
  uses it for the endpoint.
- The default endpoint security group allows ingress from the VPC CIDR block to
  TCP port 443.
- If `securityGroupIds` is provided, AWSX does not create an endpoint security
  group and does not add ingress rules.
- If `subnetIds`, `subnetConfigurations`, `privateDnsEnabled`, or
  `securityGroupIds` are provided, AWSX preserves those values.

The Auto security group default is intentionally scoped to common HTTPS AWS
service endpoints. Users who need a different endpoint port or narrower source
rules should provide `securityGroupIds` explicitly or create the lower-level
endpoint after the VPC.

Private DNS requires compatible VPC DNS attributes. In Auto mode:

- If `privateDnsEnabled` is omitted or explicitly `true`, AWSX should set
  omitted `enableDnsHostnames` and `enableDnsSupport` to `true`.
- If either DNS attribute is a literal `false` while private DNS is enabled,
  AWSX should fail early with a clear error.
- If either DNS attribute is an unknown `Input<boolean>`, AWSX should pass the
  value through and document that AWS may reject the endpoint if it resolves to
  an incompatible value.
- `privateDnsEnabled` remains the existing plain boolean component input. Dynamic
  private DNS values are not supported on `awsx.ec2.Vpc`; users who need dynamic
  private DNS should create lower-level `aws.ec2.VpcEndpoint` resources after the
  VPC.
- If `privateDnsEnabled` is explicitly `false`, AWSX should not change VPC DNS
  attributes for that endpoint.

#### Gateway Endpoints

For specs where `vpcEndpointType` is omitted or `"Gateway"`:

- If `routeTableIds` is omitted, AWSX selects all generated VPC route table IDs.
- This includes generated Private, Isolated, and Public route tables.
- If `routeTableIds` is provided, AWSX preserves it.

Gateway endpoints add service routes to selected route tables. A single route
table cannot have multiple endpoint routes for the same Gateway endpoint service.
Auto should not try to deduplicate against separately managed endpoints outside
the component; users are responsible for avoiding overlapping Gateway endpoint
routes when mixing AWSX-managed and manually managed endpoints.

#### Other Endpoint Types

For `GatewayLoadBalancer`, `Resource`, and `ServiceNetwork` endpoint types, Auto
does not infer subnet IDs, route table IDs, security groups, or DNS settings in
this change.

Those endpoint types remain expressible through explicit `vpcEndpointSpecs`
fields. Users who need more composition can create lower-level
`aws.ec2.VpcEndpoint` resources after the VPC.

## Resource Shape

AWSX should continue to create `aws.ec2.VpcEndpoint` children from
`vpcEndpointSpecs` and should use inline endpoint fields for selected subnet
IDs, route table IDs, and security group IDs.

Existing region behavior should be preserved. Endpoint resource `region` should
continue to come from the spec's `region` when present, otherwise from the VPC's
top-level `region`.

Do not create separate VPC endpoint association resources unless the underlying
provider or AWS behavior requires it. The installed provider supports inline
`subnetIds`, `routeTableIds`, and `securityGroupIds`, and separate association
resources introduce extra child identity surface.

Auto-created endpoint security groups and ingress rules are new children. Use
stable logical names derived from the existing endpoint child name, for example:

- endpoint: existing `serviceName` child name
- endpoint security group: `${serviceName}-sg`
- endpoint ingress rule: `${serviceName}-ingress`

If the implementation cannot use `serviceName` safely for the helper child names,
choose a deterministic derived name and document it before implementation. Do
not rename existing endpoint children.

## Compatibility Risks

`vpcEndpointStrategy` is an additive schema input. Existing source code should
continue to compile.

Existing programs without `vpcEndpointStrategy` remain in Legacy mode. Their
endpoint child names, explicit inputs, and provider behavior should stay stable.

Auto is opt-in, but it can create additional resources and inputs:

- Interface endpoint specs may get inferred `subnetIds`.
- Interface endpoint specs may get `privateDnsEnabled: true`.
- Interface endpoint specs may create endpoint security groups and ingress
  rules.
- Gateway endpoint specs may get inferred `routeTableIds`.
- Omitted VPC DNS attributes may become `true` when Auto Interface endpoints use
  private DNS.

Those changes are acceptable only because users explicitly choose
`vpcEndpointStrategy: "Auto"`.

Legacy warnings are user-visible diagnostics. Keep them limited to clearly
incomplete specs so working legacy programs are not noisy.

## User Guidance

### Preserve Current Behavior

Users who already pass explicit endpoint IDs should keep their current code:

```ts
new awsx.ec2.Vpc("app", {
  vpcEndpointSpecs: [
    {
      serviceName: `com.amazonaws.${region}.s3`,
      vpcEndpointType: "Gateway",
      routeTableIds: knownRouteTableIds,
    },
  ],
});
```

No strategy is required. Legacy remains the default.

### Adopt AWSX Defaults

Users who want AWSX to use generated VPC subnets and route tables should opt into
Auto:

```ts
new awsx.ec2.Vpc("app", {
  vpcEndpointStrategy: "Auto",
  vpcEndpointSpecs: [
    {
      serviceName: `com.amazonaws.${region}.s3`,
    },
    {
      serviceName: `com.amazonaws.${region}.sqs`,
      vpcEndpointType: "Interface",
    },
  ],
});
```

For Interface endpoints, Auto creates an endpoint security group that allows TCP
443 from the VPC CIDR block when `securityGroupIds` is omitted. Users who want a
different security posture should provide `securityGroupIds` explicitly or
create lower-level `aws.ec2.VpcEndpoint` resources after the VPC.

### Manual Escape Hatch

Users who need custom security groups that depend on the same VPC can create the
endpoint outside the component:

```ts
const vpc = new awsx.ec2.Vpc("app", {});

const lambdaSecurityGroup = new aws.ec2.SecurityGroup("lambda", {
  vpcId: vpc.vpcId,
});

const endpointSecurityGroup = new aws.ec2.SecurityGroup("sqs-endpoint", {
  vpcId: vpc.vpcId,
});

new aws.ec2.SecurityGroupRule("lambda-to-sqs-endpoint", {
  type: "ingress",
  securityGroupId: endpointSecurityGroup.id,
  sourceSecurityGroupId: lambdaSecurityGroup.id,
  protocol: "tcp",
  fromPort: 443,
  toPort: 443,
});

new aws.ec2.VpcEndpoint("sqs", {
  vpcId: vpc.vpcId,
  serviceName: `com.amazonaws.${region}.sqs`,
  vpcEndpointType: "Interface",
  subnetIds: vpc.privateSubnetIds,
  privateDnsEnabled: true,
  securityGroupIds: [endpointSecurityGroup.id],
});
```

## Implementation Plan

1. Add `VpcEndpointStrategy` to schema generation.
2. Add optional `vpcEndpointStrategy` to `VpcArgs`.
3. Regenerate schema and SDKs.
4. Keep `vpcEndpointSpecs` shape unchanged.
5. Reorder VPC initialization so generated subnets and route tables are
   available before endpoint specs are materialized.
6. Preserve Legacy endpoint child names, parentage, and pass-through inputs.
7. Add narrowly scoped Legacy warnings for incomplete specs.
8. Implement Auto Interface endpoint defaults.
9. Implement Auto Gateway endpoint defaults.
10. Keep other endpoint types pass-through.

## Validation

Use Pulumi runtime mock tests for component construction:

- Legacy omits generated `subnetIds` and `routeTableIds`.
- Legacy preserves explicit `subnetIds`, `subnetConfigurations`,
  `routeTableIds`, `securityGroupIds`, and `privateDnsEnabled`.
- Legacy emits warnings for incomplete Interface and Gateway specs.
- Legacy does not warn for explicit IDs.
- Auto treats omitted `vpcEndpointType` as Gateway.
- Auto fails early when `vpcEndpointType` is an unknown `Input<string>`.
- Auto Interface defaults to generated subnet IDs, one per AZ.
- Auto Interface prefers Private, then Isolated, then Public subnets.
- Auto Interface preserves explicit `subnetIds` and `subnetConfigurations`.
- Auto Interface defaults `privateDnsEnabled` to `true`.
- Auto Interface sets omitted VPC DNS attributes to `true`.
- Auto Interface fails early for literal DNS attribute conflicts.
- Auto Interface creates endpoint security group and VPC CIDR ingress when
  `securityGroupIds` is omitted.
- Auto Interface does not create endpoint security group or ingress when
  `securityGroupIds` is provided.
- Auto Gateway defaults to all generated route table IDs.
- Auto Gateway preserves explicit `routeTableIds`.
- Other endpoint types remain pass-through.

Use schema and SDK validation for public surface:

- Run schema generation.
- Inspect generated SDK diffs.
- Run schema-tools comparison against the target branch schema.
- Confirm the public surface change is additive.

Use a targeted live AWS acceptance test for the Auto happy path. Prefer
DynamoDB for the Gateway endpoint and SQS for the Interface endpoint:

- Create an AWSX VPC with isolated subnets and no NAT path.
- Configure a DynamoDB Gateway endpoint with `vpcEndpointStrategy: "Auto"`.
- Configure an SQS Interface endpoint with
  `vpcEndpointStrategy: "Auto"`.
- Create a DynamoDB table and an SQS queue.
- Deploy a Lambda function into the isolated subnet.
- Invoke the Lambda from the test.
- Have the Lambda call DynamoDB and SQS through normal regional service DNS.
- Assert both calls succeed.
- Inspect route table and endpoint state as needed to prove the generated route
  table IDs, subnet IDs, private DNS, and security group wiring are present.

Do not run the full AWS-backed test suite locally.

## Resolved Decisions

- The live acceptance test should use a DynamoDB Gateway endpoint and an SQS
  Interface endpoint. This avoids S3 global bucket naming and cleanup while
  still validating both Gateway and Interface endpoint traffic from an isolated
  subnet.
- Legacy diagnostics should use `pulumi.log.warn`, matching the existing VPC
  warning pattern.
- Auto should not expose helper child outputs in this change. Endpoint security
  groups and ingress rules remain implementation details. Users who need custom
  same-VPC security group wiring should use the lower-level endpoint escape
  hatch after constructing the VPC.
